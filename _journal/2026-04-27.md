### `MaxTextureSize=0` migration (12:39 → 13:57)

Discovered while testing the slider that `MaxTextureSize=4096` (the previous setting) caused **non-square atlases to downscale unevenly**. Idle atlases (8192×8192) downscaled by 2× to 4096×4096, but walk atlases (8192×12288) downscaled by 3× on Y to 4096×4096 — walk sprites suddenly look blurrier than idle. Fix: set `MaxTextureSize=0` (no cap) project-wide so the mip pyramid handles resolution uniformly.

Tried project-wide first, editor hung. Split:
- `clear_maxsize_01_players_a.py` through `clear_maxsize_07_enemies_03.py` (one slice each — players A/B, weapons, hair/headgear, enemies 01/02/03)
- Then enemies needed further splitting into 04a/04b through 12a/12b — the enemy folder is the biggest by count (190+ folders × 17 anims each)

Each script is idempotent (skip-if-already-correct check), safe to interrupt. Verified by `migrate_test_verify.py` later in the day.

### `SpriteCharacterActor` Path C deferred equipment swap + helmet flicker (16:11 / 16:17)

`NeverStream=False` introduces a new pop-in failure mode: equipping a weapon whose atlas is still streaming would briefly show a half-loaded or transparent texture. Path C fixes this:

- `FSpriteLayerState::PendingLayerAtlasRegistry` (TMap), `PendingSwapTexture`, `PendingSwapTimeoutSeconds`
- New `FinalizeEquipmentSwap(FSpriteLayerState&, ESpriteLayer)` method
- `LoadEquipmentLayer` now resolves the new equipment into the *pending* registry. If the new texture is still streaming, the old equipment stays visible. `Tick` polls pending swaps; finalizes when streaming completes OR 1s timeout fires.

**Critical fix during this session — helmet flicker:** `LoadEquipmentLayer` was creating a fresh `UMaterialInstanceDynamic*` on every call. Opening the inventory window caused a `LoadEquipmentLayer` re-call → new MID → one-frame flash where the new MID hadn't been bound yet. Added `if (!IsValid(L.MaterialInst))` guard so MIDs are only created once per layer-state instance. Removes the flash.

### sRGB diagnosis + canonical settings doc (20:09 → 20:21)

After the 04-27 morning import run, four enemy atlases (ambernite, parasite, plasma, rafflesia) rendered as **invisible sprites** in-game. Built `check_srgb.py` to diff settings against working references (poring, fabre):

```
fabre_idle:           srgb=True   compression=BC7  ...
ambernite_idle_wobble: srgb=False  compression=BC7  ...
```

Root cause: the body sprite material's `TextureSampleParameter2D` declares Sampler type=`Color` (sRGB-expecting). When `srgb=False`, UE5 logs `"Sampler type is Color, should be Linear Color"` and the texture is sampled as all-zeros — invisible. The original 2026-04-27 morning canonical rule incorrectly listed `srgb=False` (carried over from the old "linear color" intuition); corrected to `srgb=True` once the four enemies showed the failure mode. ## Key Decisions

- **`srgb=True` is mandatory for sprite atlases** — counter-intuitive (sprites are color-precise pixel art, the conventional wisdom says "sprites = linear"), but the body sprite material samples as Color/sRGB so the texture must be marked Color-space too. Painful lesson that took 4 invisible enemies to surface. Locked into the canonical settings table + every memory and prompt that references atlas settings. Anti-regression: `migrate_test_verify.py` checks `srgb=True` on every test atlas
- **Split mass migrations into batches of ~150-250 atlases** — UE5 editor hangs on bigger sweeps. Each script does one slice, is idempotent, can resume after a hang. Pattern formalized into the 12-script `clear_maxsize_*` and 18-script `enable_streaming_*` families
- **`MaxTextureSize=0` is canonical** — fixed caps cause non-square atlases to downscale unevenly. Walk atlases (8192×12288) downscaled by 3× on Y, idle atlases (8192×8192) downscaled by 2× — walk visibly blurrier than idle when the user looks at the same character. With 0, the mip pyramid handles uniformly
- **Path C is the right deferred-swap policy** — alternatives (block on streaming completion / show transparent during load / show fallback texture) were all worse. Keeping the *old* equipment visible until the new one is fully streamed is exactly the kind of "happy default" RO Classic players expect
- **Helmet flicker was a guard, not a refactor** — adding `if (!IsValid(L.MaterialInst))` is one line; refactoring `LoadEquipmentLayer` to be re-entrant or stateless would have been a half-day. The guard is correct, the rest is over-engineering
- **Despawn high-level enemies in `prt_south` for VRAM testing, don't delete the spawn data** — commented out, with a clear "TODO restore after testing" marker. Restoring is one find-and-uncomment
