## What Got Done

### Spawn Region System — UE5 placeable actors + JSON export + server loader (13:51 → 14:08)

**Two new placeable actors** in the UE5 editor (`client/SabriMMO/Source/SabriMMO/SpawnRegionVolumes.{h,cpp}`):

- `ASpawnAllowVolume` — green wireframe `UBoxComponent`. Defines where enemies are *allowed* to spawn.
- `ASpawnDenyVolume` — red wireframe `UBoxComponent`. Carves out zones (warp portals, NPC spots, player spawn) where enemies must *not* spawn even if inside an overlapping allow box.

Each volume has two `UPROPERTY` fields:
- `MonsterFilter: TArray<FString>` — whitelist of template names. Empty = all templates allowed.
- `RegionTag: FString` — free-form label written to the export JSON for debugging.

**JSON exporter** (`SpawnRegionExporter.{h,cpp}`):
- New console command: `ExportSpawnRegions <ZoneName>`
- Walks every `ASpawnAllowVolume` / `ASpawnDenyVolume` in the loaded level
- Writes their world-space AABBs (via `UBoxComponent::Bounds`) + tag + filter to `server/spawn_regions/<ZoneName>.json`
- Rotated boxes export as the smallest enclosing AABB (slightly looser than the rotated shape, but always respects the broad area)

**Server loader** (`server/src/ro_spawn_regions.js`):
- `initSpawnRegions()` — called once at server boot, eagerly loads every JSON in `server/spawn_regions/`
- `generateSpawnsFromPool(zoneName, spawnPool)` — picks weighted-random points from allow boxes (weight ∝ XY area), rejects points inside any overlapping deny box, snaps to nearest NavMesh point via the existing `findClosestNavMeshPoint` (500 UU halfExtent on Z so vertical box positioning doesn't matter)
- `hasSpawnRegions(zoneName)` — gate so zones without an export skip silently

**Documentation**: `docsNew/05_Development/Spawn_Region_System.md` (16,852 bytes) covers the full pipeline — UE5 placement guide, MonsterFilter examples, weighting math, console command, JSON schema, server architecture diagram.

**`enemySpawns[]` and `spawnPool[]` coexist**: a zone can use both. `enemySpawns[]` runs first (hand-placed exact coordinates), then `spawnPool[]` adds random spawns on top. Designed for the migration path — keep existing precise spawns, add random density layered on top.

## Key Decisions

- **JSON export is one-shot, not live** — running `ExportSpawnRegions` writes a snapshot. Designer iterates: edit boxes → export → restart server. No live editing of placed volumes (would need a custom WebSocket bridge — over-engineering for the current need).
- **Weight by XY area, not count** — a 4000×4000 allow box gets ~16× the spawns of a 1000×1000 box. Lets designers concentrate spawns near corridors without manually scaling counts.
- **NavMesh snap on every random point** — even if the box accidentally extends below the floor or hovers above. The 500 UU halfExtent on the up axis covers any reasonable vertical extent.
- **Box `MonsterFilter` is a whitelist, not blacklist** — easier to reason about: "this box only spawns Porings and Fabres" reads cleanly, "this box spawns everything except X and Y" is awkward.
- **Hunyuan3D `mc_algo="mc"` is mandatory** — `dmc` looked attractive in the docs but is broken on the RTX 5090 / torch 2.12 / sm_120 stack. Lock this in everywhere from day 1.
- **Pre-mask BEFORE Hunyuan3D** — letting the Hy3D pipeline auto-bg-remove strips green canopies. rembg with BiRefNet preserves them. Small thing, big visual difference.
