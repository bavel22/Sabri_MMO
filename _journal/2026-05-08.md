### `ZoneTransitionSubsystem` preload-wait (02:02)

Critical fix for the 04-26/04-27 sprite-quality slider work. Without this, the loading screen dismissed while sprite atlases were still streaming → pop-in storm on zone entry.

`ZoneTransitionSubsystem.h` adds:
```cpp
FTimerHandle PreloadWaitTimer;
double PreloadWaitStartTime = 0.0;
double PreloadStableSinceTime = 0.0;

void WaitForPreloadAndHideOverlay();
void ShowExpectedZoneChange(const FString& StatusText);
void HideExpectedZoneChange();
```

**`WaitForPreloadAndHideOverlay`** — replaces the previous "next-tick HideLoadingOverlay" flow. Polls `UZonePreloadSubsystem::IsLoadingInProgress()` every 200ms with these conditions:

1. If any in-flight loads → reset stability timer, keep waiting
2. If zero in-flight → mark stable, wait for AT LEAST 700ms of stability AND 1.5s since wait started (so the zone:ready → enemy:spawn network round-trip has time to enqueue more loads)
3. If 15s elapsed total → hard timeout, dismiss anyway with warning log

Both `CheckTransitionComplete` (normal completion) and `ForceCompleteTransition` (pawn-teleport-failed fallback) now hand off to `WaitForPreloadAndHideOverlay` instead of next-tick dismissal.

**`ShowExpectedZoneChange` / `HideExpectedZoneChange`** — public API for code paths that *expect* a zone change (Kafra teleport, Butterfly Wing, RequestWarp) to optimistically show the overlay before the server response arrives. `HandleZoneError` calls `HideLoadingOverlay` to dismiss when the server rejects.

`RequestWarp` now also shows the overlay immediately for responsive feedback. The actual zone change re-shows it with the destination display name.

## Key Decisions

- **Loading screen waits for preload, with a 15s timeout** — better to make zone transitions take an extra 1-3 seconds than to dump the player into a half-streamed atlas storm. The 15s hard cap prevents trapping if preload truly hangs (network failure, missing manifest).
- **700ms stability + 1.5s minimum wait** — the round-trip "zone:ready emitted → enemy:spawn arrives → preload begins" is variable. 1.5s minimum gives the network time to enqueue; 700ms of zero-in-flight ensures we're not just between batches.
- **Optimistic overlay on RequestWarp / Kafra / Butterfly** — players notice 200-300ms of "did my click register?" otherwise. Show overlay immediately, dismiss on `zone:error` if the server rejects, otherwise the normal flow re-shows + dismisses correctly.
- **Per-zone presets in `ApplyZonePreset`, NOT a Material Parameter Collection** — current scale is fine for if/else. Will revisit if zones balloon to 20+.
- **Directional shadow cap = 7000 UU** — empirically the largest distance where the 15° FOV doesn't show cascade seams at the screen edges. Below 5000 the foreground starts to clip.
- **Sewer fog density 0.30 with falloff 0.2** — needed to push fog to player altitude. The default falloff 2.0 makes density drop near-zero before reaching the capsule (sewer floor sits at Z~620, player capsule top is at Z~720). Shallow falloff 0.2 distributes density more evenly.
- **Single SpawnAllowVolume covering the whole footprint + single deny carving canal water** — simplest valid topology. Future iterations can split the allow into per-room volumes with different `MonsterFilter` arrays for thematic spawning.
- **AWarpPortal radius 250 not 200** — cramped indoor geometry leaves less margin than a field zone. 250 catches walking-in-from-corner approaches.
- **`new_zone_creation.md` as the canonical bootstrap** — same self-contained re-entry pattern as the other big system prompts. SewerDungeon01 is the worked example.
