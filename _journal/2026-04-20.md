### Client C++ — EnemySubsystem Hardening

`EnemySubsystem.h`:
```cpp
FLinearColor SpriteTint = FLinearColor::White;  // Body tint multiplier
bool bCanMove = true;                            // false = stationary (no attack lunge)
```

`EnemySubsystem.cpp::HandleEnemySpawn` — new parse block between WeaponMode read and entry-create:
- `spriteTint` (3-element JSON array) → `FLinearColor`
- `size` ("small"|"medium"|"large") → 0.5/1.0/1.5 scale
- `spriteScale` (number) → overrides size scale
- `canMove` (bool, default true) → stored on entry

Applied via `SetLayerTint(Body, …)` + `SetActorScale3D(FVector(Scale))` after sprite spawn. Re-apply tint on respawn path (idempotent when MID survives).

`HandleEnemyAttack` — new `bCanMove` gate on lunge:
```cpp
if (Entry->bCanMove && ActualLunge >= 5.f && DistToTarget <= MeleeThreshold) { /* lunge */ }
```
Stationary enemies (plants / crystals / eggs) still play the attack animation but stay rooted — they thrash in place instead of snapping forward.

Ranged attack (`DistToTarget > MeleeThreshold`): vine decal swap. Old code spawned an `MI_T_Decal_RootTendril` at target feet (invisible from angled camera). New code spawns a billboard `ASpriteCharacterActor` at target feet using the attacker's own sprite class, scaled 0.6x, Attack anim, `TWeakObjectPtr` destroy timer at 0.8s. Visible from every angle.

### render_monster.py + blender_sprite_render_v2.py

New `--model-z-offset` flag — lifts mesh by N Blender units before rendering. Preserves during lunge interpolation and animation resets. Intended use: flying creatures (bats, ghosts, airborne blobs) want `--model-z-offset 0.5`.

Default lighting bumped from `--cel-shadow 0.45 --cel-mid 0.78` (dark, high-contrast) to `--cel-shadow 0.92 --cel-mid 0.98` (soft posterized RO Classic). Inline comment references `feedback-render-pipeline-standard.md`.

### pupa_v2.json

Added walk + attack state fallback — reuses `Idle Wobble` folder. The egg preset generates 3 anims (idle / hit / death) but the client animation state machine still transitions to walk/attack on certain events. Mapping both to the idle loop avoids "missing animation" flash.

## Key Decisions

- **Grid spawn layout, not ad-hoc** — spawns go in rows at predictable Y offsets (±2000 between batch rows, ±200 within a batch). Makes manual QA sane: walk south from (0,0) and you pass every batch in order. New-enemy debug doesn't need to hunt
- **Variant inheritance is free coverage** — stop rendering `meta_X` separately. Set `meta_X.spriteClass = 'X'` in the template and move on. The enum script already recognizes the rule
- **Steel Chonchon = own model, not a tint** — the original 04-15 plan had Steel Chonchon in the recolor candidates list. Today rendered it as a unique model anyway since we had the GLB. Tint system is still exercised for Eclipse (Lunatic) + plasma variants (plasma_r/b later in the week)
- **`spriteScale` is a separate field from `size`** — `size: 'large'` is the RO template's authoritative size bucket (drives attack range, knockback resistance, etc.). `spriteScale` is purely visual, letting us have `whisper_boss` read as 1.5× regular whisper without messing with size-gated combat math
- **Ranged attack = billboard, not decal** — ortho camera means XY decals are invisible at most angles. Billboards always face camera. Added memory about this previously; today it's implemented
- **`canMove` derived from `modeFlags.canMove`, not a new top-level field** — server already tracks mode flags (aggressive / assist / passive / etc.); `canMove: false` is just another mode on plants / crystals / eggs. Zero new schema pressure
- **Rotation 90 for egg/plant preset, -90 for everything else, -75 for thin wings** — locked in as the three standard orientations. Anything else gets called out (`--model-rotation` override per-render)
