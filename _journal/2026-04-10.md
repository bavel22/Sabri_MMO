### Inventory Performance Analysis (root cause report)
- Created `_journal/inventory_performance_analysis.md` — comprehensive 10-finding root cause report
- **Server findings** (4): Full 62-column DB fetch on every change (53 call sites), description bloat in payload (~164KB for 200 items), weight recalc via separate DB query every mutation
- **Client findings** (6): Full widget rebuild every update (350 widgets destroyed/created), GetItemAtSlot copies entire filtered array per-frame (~360 MB/sec garbage), GetFilteredItems returns by value, no TMap lookup (O(N) linear search), hotbar polls 36 x O(N) every second, tooltips pre-built during rebuild
- **4-phase optimization plan**: Phase 1 (cache + references), Phase 2 (delta widget update), Phase 3 (strip descriptions from payload), Phase 4 (incremental weight + itemDef Map)

### Inventory Optimizations Started (Phase 1 partial)
- `GetFilteredItems()` changed to return `const TArray<FInventoryItem>&` (reference, not copy)
- Added `RebuildFilteredCache()` for cached filtered items
- Added `ItemDefCache` TMap for static item data by item_id
- Added `HandleItemDefs()` handler for `itemDefs:data` socket event
- Added `HandleRefineResult()` handler for `refine:result` event

## Key Decisions

- **Wrote analysis report before optimizing** — the cascade effect (2 DB queries + 164KB transfer + 350 widget rebuilds for 1 potion) justified a structured 4-phase plan rather than piecemeal fixes
- **Phase 1 first** (client caching) — highest impact, zero risk, eliminates the per-frame hemorrhaging before touching server code
- **Return by const reference** instead of by value for GetFilteredItems — eliminates hundreds of MB/sec in temporary allocations
