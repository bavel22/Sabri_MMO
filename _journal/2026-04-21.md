## Key Decisions

- **Batch scripts as normal workflow** — every new batch gets a `.add_batchN.js` next to the `_v2.json` configs and the `import_batchN.py`. The JS scripts are idempotent (skip-if-already-has-spriteClass) so rerunning is safe. This pattern now clearly scales; expect to use it for every future batch
- **Plasma gets 4 tint siblings, not 4 atlases** — `plasma` base atlas is rendered once; `plasma_r` / `plasma_b` / `plasma_g` / `plasma_p` are template-level tint overrides. Saves 3× Blender time per variant, plus validates the tint system with more than just Eclipse
- **`blazzer` intentionally skipped then re-added same day** — when the batch 11 run started the GLB wasn't on disk. Explicit skip in the JS kept the batch atomic. When the GLB arrived hours later, a one-off `.add_blazzer.js` + render + re-run of `import_batch11.py` closed the loop without re-running the full batch
- **Late additions don't get renumbered batches** — `vocal`, `blazzer`, `orc_baby` are "orphan" mini-batches that piggyback on existing batch import scripts. Avoids batch-number inflation and keeps the overall sequencing readable
- **`wild_ginseng` handled via variant inheritance, not its own atlas** — `wild_ginseng` spawns use `hermit_plant` sprite (from 04-20). Server template inherits; client never needs to know it's a recolor. Zero render cost, zero C++ change
- **200u horizontal × 200u vertical grid stays** — matches 04-20's spacing. Every future batch row slots into the same 12×N grid, which makes "walk the spawn gallery" QA trivial
