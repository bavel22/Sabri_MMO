### Sprite-quality system architecture (late evening, 23:04 → 23:42)

This is the structural prep for the runtime sprite-quality slider — slider itself ships from 04-27 01:56. Tonight is the **preload pipeline**.

- **`ZonePreloadSubsystem.h/.cpp`** (created) — per-world subsystem. Three handle tiers:
  1. **PinnedHandles** — local player class always pinned
  2. **ActiveHandles** — current zone + adjacent classes
  3. **LRU cache** — 4 GB budget
  Public API: `RequestClassPreload(SpriteClass)`, `RequestLayerPreload(Layer, ViewSpriteId, GenderSubDir)`, `PinClass`, `BeginZone(NewZoneName)`, `SetLocalPlayerClass`, `IsLoadingInProgress`, `OnAllPreloadsComplete` delegate, `GetInFlightCount` / `GetResidentClassCount` / `GetApproxResidentBytes` for debug overlay. Async on `FStreamableManager` + `FStreamableHandle`.

- **`EnemySubsystem.cpp` HandleEnemySpawn`** — added a 9-line block that grabs the world's `UZonePreloadSubsystem` and calls `RequestClassPreload(SpriteClass)` before spawning. Idempotent — 50 porings spawning back-to-back trigger one batch load. Sprite spawns immediately; texture data resolves from cache once async completes.

- **`OtherPlayerSubsystem.cpp` HandlePlayerMoved`** — bigger 60-line block: parses `characterName` / `jobClass` / `gender` / `hairStyle`, builds the remote sprite class string (`{job}_f` or `{job}_m`), preloads body, hair, and the 6 equipment slots (`weapon` / `shield` / `head_top` / `head_mid` / `head_low` / `garment`) if present in the `equipVisuals` payload. Earlier than `HandlePlayerAppearance` would do it — gives the streamer a head start.

- **`ZoneTransitionSubsystem.h/.cpp`** — wired the loading screen to wait on `IsLoadingInProgress() / OnAllPreloadsComplete`. Loading screen no longer hides until preload says it's done. (Or until a hardcoded timeout — confirm in code if loading screens start hanging.)

- **`server/src/index.js` zone:ready handler** — added a 30-line block (around the existing zone:ready emit). Walks `ZONE_REGISTRY[zone].warps`, builds an `adjacentClasses` object keyed by `destZone` whose values are arrays of distinct, non-dead `enemy.spriteClass` for that zone. Emits `zone:adjacent_classes { zones: { ... } }` to the joining socket so the client can warm up neighbour-zone atlases predictively. Wrapped in try/catch so a malformed `ZONE_REGISTRY` entry can't break `zone:ready`.

## Key Decisions

- **Pull the seven-day commit boundary first** — the `56dce91` commit lands before any of the day's heavier C++ work, so the tree is clean when the sprite-quality system starts. If the new architecture proves wrong tomorrow, `git reset --hard 56dce91` is a clean revert.
- **Homunculus stays as placeholder sprites** — running the 3D-to-2D pipeline for 4 dedicated homunculus sprites would be ~30-40 minutes per type given the standard workflow. Picking 4 cute existing enemies (poring/poporing/picky/drops) gives "different visual per type" today, with the full sprite render deferred to whenever someone wants polish over depth.
- **Lif-only autocast** — getting Healing Hands self-autocast for Lif covers 80% of "I never use my homunculus skills" complaints. Doing all 4 types properly requires a per-type behaviour tree, which is a much bigger design pass; ship Lif now, plan the rest later.
- **Server emits `zone:adjacent_classes`** — predictive preloading needs to know which classes the *adjacent* zones will spawn. Server has `ZONE_REGISTRY[zone].warps` + the live `enemies` Map; client doesn't. Putting the computation server-side (one walk + emit per `zone:ready`) is far cheaper than the client guessing or making a separate request.
- **Three-tier handle system in `ZonePreloadSubsystem`** — pinning the local player's class is non-negotiable (you'll always need your own atlases). Active = current + adjacent (the working set). LRU at 4 GB lets recently-visited zones survive a brief return without a reload but doesn't pin the world. The 4 GB budget is a guess — measure tomorrow.
